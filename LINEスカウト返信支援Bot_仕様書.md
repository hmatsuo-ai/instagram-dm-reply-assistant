# LINE Bot スカウト返信支援システム 仕様書

**文書バージョン**: 1.7  
**目的**: メッセージ履歴を RAG 用に構造化し、LINE Bot 経由で返信案を提示する。特定担当者の暗黙知に依存せず、一定水準の返信を誰でも再現できる状態を目指す。

---

## 1. 概要

### 1.1 システム名

LINE Bot によるスカウト返信内容の考案支援（履歴学習型 RAG）

### 1.2 目的

- スカウト担当者のノウハウに偏らず、**経験の浅い担当者やサポート要員でも、過去のやり取りに近い品質の返信案を短時間で得られる**ようにする。
- 過去に蓄積した**メッセージ履歴**を解析し、検索可能なナレッジとして保持する（RAG）。
- 利用者は LINE 上で状況や相手の文面を入力し、**返信の下書き・複数案**を受け取る。

### 1.3 スコープ

**インフラの前提**: オフィス内に **24 時間稼働させる常設 PC（以下「自営サーバ」）** を置き、LINE Webhook・RAG・LLM・`replyMessage` を **同一マシン上のアプリケーション**で実行する。クラウドのマネージド実行基盤（Cloud Run 等）は**既定では使わない**。

| 含む | 含まない |
|------|----------|
| **自営サーバ**上の HTTPS アプリによる Webhook 受信、署名検証、RAG、LLM、**`replyMessage`** | **Google Apps Script を必須とする構成**（任意補助としての利用は §3.3） |
| メッセージ履歴の取り込み・前処理・RAG インデックス構築（同一 PC または運用フローで取り込み） | 候補者へのメッセージの自動送信（いずれのチャネルも対象外） |
| LINE Bot による返信案の生成・提示 | 社外への一斉配信・マーケティング自動化 |
| 社内担当者向けの利用制限・ログ | リアルタイムの会話傍受・録音解析 |

**本システムは「返信内容を考える支援」であり、Instagram／LINE 等への送信は担当者が手動で行う。**

---

## 2. 背景とゴール

### 2.1 背景

返信の言い回しや構成は、従来、ベテラン担当者のノウハウに依存しやすく、担当者によって品質やスピードに差が出ていた。経験の浅い担当者や他部署のサポートでは、適切な返信に自信を持てず、都度確認や作業の停滞が発生しやすい。

### 2.2 ゴール

- 過去のメッセージ履歴を RAG 用に構造化し、**類似場面の「うまくいったやり取り」**を検索に利用できるようにする。
- LINE Bot が、入力内容に応じて**複数パターンの返信案**を提示する。
- **属人化を解消**し、オンボーディング期間を短縮する。

### 2.3 成功指標（例）

- 返信案生成までのリードタイム（目標値は別途合意）
- 担当者アンケートによる「そのまま使える／参考になった」比率
- エスカレーション・再確認の件数推移

## 3. システム構成

### 3.1 推奨アーキテクチャ（オフィス常設 PC ＝ 自営サーバ）

**LINE の Webhook、署名検証、RAG、LLM、Messaging API（`replyMessage`）を、オフィス内で 24 時間稼働させる 1 台の PC 上のアプリケーションに集約する。**  
クラウド上の別サーバレスは**必須としない**。

```
[メッセージ履歴] → バッチ（例: build_rag_chunks.py）→ RAG インデックス
        （開発時は別 PC 可）                    （自営サーバ上のディスク／埋め込み DB に配置）
                              ↑
[担当者 LINE] ⇄ LINE Platform ⇄ [自営サーバ: 固定 HTTPS Webhook URL]
                                    │ 署名検証・イベント処理
                                    │ 埋め込み・ベクトル検索・LLM
                                    │ replyMessage
                                    ▼
担当者の LINE に返信案を表示（コピーして Instagram 等へ手動送信）
```

- **利用者**: 社内スカウト担当者（LINE User ID によるホワイトリスト）
- **データ保存先（ナレッジ・ログ）**: **自営サーバのローカルディスク**（必要に応じてバックアップ先を別途定義。個人情報・同意に準拠）

### 3.2 コンポーネント役割

| コンポーネント | 役割 |
|----------------|------|
| **自営サーバ（常設 PC）** | **常時起動**・**電源・ネットワーク**の維持。OS は **Windows または Linux**（実装者が保守可能なもの）。HTTPS で **Webhook エンドポイント**を公開。**チャネルシークレット**で署名検証。**チャネルアクセストークン**で `replyMessage`。**RAG**（埋め込み・類似検索）、**LLM API 呼び出し**、ホワイトリスト判定、ログ出力を実装。 |
| **アプリプロセス** | 本番では **Windows サービス / systemd / 常駐プロセス＋自動再起動** 等でプロセス落ちに耐える。**replyToken の有効期限内**に処理を完了させる。 |
| **オフラインバッチ** | メッセージ履歴の正規化・チャンク化（例: `scripts/build_rag_chunks.py`）。**成果物（JSONL 等）を自営サーバが読み込む**パスを決める（共有フォルダ、手動コピー、Git、スケジュール実行など）。 |

### 3.3 インターネット到達性（必須要件）

LINE Platform は **社内 LAN のローカル IP へ直接 Webhook しない**。自営サーバには次のいずれかが必要。

- **公開 HTTPS URL**: ルータの **ポート転送**、**固定グローバル IP 又は DDNS**、**TLS 証明書**（Let's Encrypt 等）
- **トンネル／リバースプロキシ**: **Cloudflare Tunnel**、**ngrok** 等で **HTTPS の外向き URL** を自営サーバに中継（社内 FW でポート開放が難しい場合に有効）

### 3.3.1 Cloudflare Tunnel 手順（推奨運用例）

本節は、社内ネットワークのポート開放が難しい場合の最短手順である。

1. **Cloudflare Zero Trust** でトンネルを作成し、Connector（`cloudflared`）を自営サーバへ導入する。  
2. 自営サーバ上のアプリを起動する（例: `python -m uvicorn bot_server.main:app --host 0.0.0.0 --port 8000`）。  
3. トンネルの Public Hostname を設定し、サービス先を `http://localhost:8000` に向ける。  
4. 発行された `https://<公開ドメイン>` で `GET /health` が到達することを確認する。  
5. LINE Developers の Webhook URL に以下を設定する。  
   - 直接Webhook構成: `https://<公開ドメイン>/webhook/line`  
   - GAS中継構成: GAS Webアプリ URL（`/internal/suggest-replies` は GAS から呼び出す）  
6. LINE Developers で Webhook URL を `Verify` し、`Use webhook` を有効化する。  
7. 自営サーバ再起動時にトンネルが自動復旧するよう、`cloudflared` をサービス化する。  

運用上の注意:

- Tunnel 認証情報は自営サーバ内で保護し、リポジトリへ保存しない。  
- ドメイン切替時は LINE Developers の Webhook URL も同時更新する。  
- DNS 反映直後は `Verify` が不安定になる場合があるため、数分待って再試行する。  

### 3.4 ハードウェア・運用の前提

| 項目 | 内容 |
|------|------|
| 稼働 | **24 時間電源投入**。停電対策として **UPS** を推奨 |
| 再起動 | OS 更新後の **自動起動**、アプリの **常駐・自動復帰**（タスクスケジューラ／サービス等） |
| 監視 | ディスク容量、プロセス生存、外部からの Webhook 疎通（将来） |
| セキュリティ | OS 更新、不要ポートの閉塞、管理画面の認証、機械の物理セキュリティ |

### 3.5 Google Apps Script（推奨：秘密情報の置き場所）

**LINE のチャネルシークレット・チャネルアクセストークンは、Google Apps Script の「スクリプト プロパティ」に登録し、リポジトリ・自営サーバの `.env` には書かない**運用を推奨する。

| 秘密・設定 | 置き場所 | 備考 |
|------------|----------|------|
| `LINE_CHANNEL_SECRET` | GAS スクリプトプロパティ | Webhook 署名検証（ヘッダが取れない実行環境では §5.9 のクエリトークン併用） |
| `LINE_CHANNEL_ACCESS_TOKEN` | GAS スクリプトプロパティ | `replyMessage` のみ GAS が実行するためサーバ不要 |
| `BACKEND_SUGGEST_URL` | GAS スクリプトプロパティ | 例: `https://（トンネル）/internal/suggest-replies` |
| `INTERNAL_WEBHOOK_SECRET` | **GAS と自営サーバの `.env` に同一値** | GAS が `Authorization: Bearer` で自営 API を呼ぶ。**プロパティと環境変数のみ**にしコードに埋め込まない |
| `WEBHOOK_QUERY_TOKEN`（推奨） | GAS スクリプトプロパティ | Web アプリ URL に `?token=...` を付し、GAS で照合（URL 秘匿の補強） |
| `OPENAI_API_KEY` | 自営サーバ `.env` のみ | LLM は自営で実行。GAS 経由でキーを渡さない |
| `LINE_ALLOWED_USER_IDS` | GAS プロパティ（任意） | カンマ区切り。空なら全許可（非推奨） |

GAS 実装例: リポジトリ `gas/LineRelay.gs`。RAG／LLM は引き続き **自営サーバ**上で実行する。

## 5. メッセージ履歴の解析と RAG 用構造化

### 5.1 想定ソース形式

Instagram の「情報のダウンロード」等で得られる **JSON** を想定する。  
`messages/inbox/<会話フォルダ>/message_*.json` などに、会話単位のファイルが格納される。

**参照スキーマ例（抜粋）**:

- `participants`: 参加者名の配列
- `messages[]`: メッセージ配列
  - `sender_name`
  - `timestamp_ms`
  - `content`（本文）
- `title`, `thread_path` 等

バージョンによりキー名や追加フィールドがあるため、**取り込み時にスキーマ検証**を行う。

### 5.2 正規化レイヤ

| 処理 | 内容 |
|------|------|
| 文字コード | UTF-8 に統一。エスケープされた Unicode を通常文字に復元 |
| 会話 ID | `thread_path` またはファイルパスから一意な `conversation_id` を付与 |
| 発話単位 | `messages[]` の各要素を 1 レコードとして扱う |
| 送信者区分 | `sender_name` が自社アカウント名（または設定ルール）と一致するかで `role`: `business` / `candidate` / `unknown` を付与 |
| 時刻 | `timestamp_ms` を ISO8601 等に変換し `sent_at` として保持 |

### 5.3 チャンク設計（RAG 用）

検索精度と文脈のバランスのため、次のいずれかまたは併用を定義する。

| チャンク種別 | 単位 | 用途 |
|--------------|------|------|
| C-1 | 単一メッセージ | 短い定型・単発 Q&A の類似検索 |
| C-2 | スライディングウィンドウ | 直近 **N 往復**（例: 5〜10 メッセージ）を 1 チャンク。文脈付きの返信パターン検索 |
| C-3 | スレッド要約（任意） | 長会話の要約を別チャンクとして保持し、メタ検索に利用 |

各チャンクに付与する**メタデータ（例）**:

```json
{
  "chunk_id": "uuid",
  "conversation_id": "string",
  "chunk_type": "single_message | window | summary",
  "sent_at": "2025-01-01T12:00:00+09:00",
  "role": "business | candidate | unknown",
  "text": "検索・埋め込み用の本文（マスク後）",
  "source_file": "相対パス（内部管理用）"
}
```

### 5.4 個人情報・マスキング

- 必要に応じて**電話番号・メール・特定の固有名詞**をマスクした `text` を埋め込み対象にする。
- マスク済み／未マスクの扱いは**データ区分**で管理し、社内規程に合わせる。

### 5.5 インデックス

- **ベクトル**: チャンク `text` を埋め込みモデルでベクトル化し、**自営サーバ上**のベクトル DB またはファイルベース索引に保存（SQLite+拡張、Chroma ローカル、faiss ファイル等。実装時に選定）。
- **オプション**: ハイブリッド検索のため、キーワード索引（BM25 等）を同一アプリ内で併用。
- **インデックスの所在**: クラウド必須ではなく、**自営サーバのローカル**を既定とする。

### 5.6 検索〜生成フロー

フローは **自営サーバのアプリ内**で完結する（Webhook 受信から `replyMessage` まで同一プロセスまたは同一ホスト内）。

1. 担当者の入力（クエリ）を埋め込み、類似チャンクを **top-k** 取得（フィルタ: 日付、role 等）。
2. プロンプトに「クエリ + 取得チャンク（引用として明示）」を渡し、LLM が返信案を生成。
3. 出力に**参照したチャンク ID** をログに残す（説明可能性・監査用）。

### 5.7 バッチ実装（リポジトリ内）

| 項目 | 内容 |
|------|------|
| スクリプト | `scripts/build_rag_chunks.py`（標準ライブラリのみ） |
| 設定 | `config/rag_business_patterns.json` … `sender_name` に含まれる部分文字列で `role=business` と判定 |
| 出力 | `output/rag_chunks.jsonl`（1 行 1 チャンク）、`output/rag_build_summary.json`（件数サマリ） |
| 取り込み対象 | `メッセージ履歴/messages/inbox` 配下の **サブフォルダ内 `message_*.json`** および **inbox 直下の `*.json`** |
| チャンク種別 | `single_message`（各メッセージ）、`window`（最大 `--window-size` 通、開始位置を `--window-stride` でずらし） |
| 文字化け対策 | Meta エクスポートで UTF-8 バイトが誤った Unicode として格納されている場合、Latin-1 経由で UTF-8 に再解釈する処理を適用 |

**実行例**

```text
python scripts/build_rag_chunks.py
python scripts/build_rag_chunks.py --window-size 12 --window-stride 4 --max-files 20
```

### 5.8 履歴の整理（一方的スレッドの除外）

RAG には「やり取り」の方が有用なため、**候補者（`role=candidate`）のメッセージが 1 件もない**スレッドを `inbox` から除き、アーカイブへ移す。

| 項目 | 内容 |
|------|------|
| スクリプト | `scripts/prune_one_sided_threads.py` |
| 判定 | `config/rag_business_patterns.json` と同一ルールで送信者区分。**いずれかのメッセージで candidate と判定されれば維持**、1 件もなければ一方的とみなす |
| 移動先 | `メッセージ履歴/messages/inbox_archived_one_sided/folders/`（元がサブフォルダの場合）、`.../root_json/`（inbox 直下の単一 JSON） |
| 既定 | 引数なしは **dry-run**（件数のみ）。`--execute` で移動 |

```text
python scripts/prune_one_sided_threads.py
python scripts/prune_one_sided_threads.py --execute
```

整理後は `build_rag_chunks.py` を再実行して `rag_chunks.jsonl` を更新する。

### 5.9 ランタイム実装（`bot_server` + `gas`）

| 項目 | 内容 |
|------|------|
| 言語・FW | Python 3、`FastAPI` + `uvicorn` |
| 依存 | `requirements-bot.txt`（`pip install -r requirements-bot.txt`） |
| **GAS 中継（推奨）** | `gas/LineRelay.gs` をプロジェクトにコピーし、ウェブアプリとしてデプロイ。LINE の Webhook URL は **GAS の URL**（オプションで `?token=WEBHOOK_QUERY_TOKEN`）。GAS が `POST /internal/suggest-replies` を `Authorization: Bearer INTERNAL_WEBHOOK_SECRET` で呼び、戻り `text` を `replyMessage` |
| 内部 API | `POST /internal/suggest-replies` … JSON `{"user_text":"...","line_user_id":null}`、応答 `{"text":"...","cited_chunk_ids":[...]}`。`INTERNAL_WEBHOOK_SECRET` 未設定時は **404（無効）** |
| 直接 Webhook（任意） | `POST /webhook/line`。**`ALLOW_DIRECT_LINE_WEBHOOK=false`** で無効化し GAS のみにできる |
| `GET /health` | 疎通、`internal_api_enabled` 等 |
| RAG 検索 | `output/rag_chunks.jsonl` を起動時に読み込み、**TF-IDF + コサイン類似度**で top-k |
| LLM | 自営 `.env` の `OPENAI_API_KEY`（GAS には置かない） |
| 自営 `.env`（GAS 併用時） | `INTERNAL_WEBHOOK_SECRET`（必須）、`OPENAI_API_KEY`、`ALLOW_DIRECT_LINE_WEBHOOK=false` 推奨。**`LINE_CHANNEL_*` は空でよい** |
| 起動例 | `python -m uvicorn bot_server.main:app --host 0.0.0.0 --port 8000`（外向き HTTPS は §3.3） |

### 5.10 GAS での秘密管理の注意

- **スクリプト プロパティ**の編集権限を最小限にし、ソースにベタ書きしない。
- プロパティは**組織内の閲覧権限**があるロールでも参照できる場合があるため、アカウント権限設計と合わせる。
- `INTERNAL_WEBHOOK_SECRET` は十分に長い乱数（例: 32byte 以上を hex 化）を使い、**自営サーバの環境変数と同値**にする。

## 6. LINE Bot 連携（自営サーバ）

### 6.1 技術要件

- LINE Messaging API（公式アカウント）
- **Webhook URL**: **推奨: GAS ウェブアプリの URL**（`gas/LineRelay.gs`）。直接自営へ繋ぐ場合は `https://（公開ホスト）/webhook/line`
- 自営サーバ: 受信ボディに対する **署名検証**（`X-Line-Signature` とチャネルシークレット）
- 同一プロセス（又は同一ホスト）で **RAG + LLM** を実行し、**`replyMessage`** で応答（チャネルアクセストークンは環境変数・ローカルシークレット等で保管）。長文は分割送信を検討
- **replyToken の有効期限内**に処理を完了できなければ失敗するため、LLM タイムアウト・モデル選択・top-k を設計する（§7）

### 6.2 利用フロー（例）

1. 担当者が LINE で Bot にメッセージ送信（例: 「候補者から『体験の時間は？』と来た」）。
2. **LINE → 自営サーバ** で Webhook 受信。署名検証後、テキストと `replyToken`・`userId` を取り出す。
3. アプリが **RAG + LLM** で返信案を生成する。
4. **`replyMessage`** で LINE に返信案を表示。担当者がコピーし、**Instagram 等で手動送信**。

### 6.3 内部モジュール境界（API 形の整理）

単一プロセス内でも、実装の読みやすさのため **「検索」「LLM」「LINE Webhook ハンドラ」**に分離してよい。マイクロサービス化は**必須ではない**。  
複数プロセスに分ける場合は **同一 PC 上のローカル HTTP**（例: `127.0.0.1`）に限定し、インターネットに公開するのは Webhook 用エンドポイントに集約する。

### 6.4 自営サーバ上の制約

- **CPU・メモリ（・GPU）**: 埋め込み・LLM を **ローカル推論**にするか **外部 API** にするかで要件が変わる。外部 API の場合は **外向き HTTPS（プロキシ・FW 許可）** が必要。
- **replyToken 失効**: 処理遅延時は定型の短い返信やエラー文言の方針を定める。
- **OS メンテナンス**: Windows Update 等の再起動と **自動復帰**手順を定義。

### 6.5 レート制限・利用制限

- 同一利用者の連続送信に対するクールダウン（任意・アプリ内）
- ホワイトリスト外の User ID からのイベントは **Webhook ハンドラで弾く**か、定型文のみ返信

### 6.6 Webhook URL 設定手順（Cloudflare Tunnel 前提）

1. LINE Developers コンソールで対象チャネルを開く。  
2. `Messaging API` タブの `Webhook URL` に以下を入力する。  
   - 直接Webhook構成: `https://<公開ドメイン>/webhook/line`  
   - GAS中継構成: `https://script.google.com/.../exec?token=<WEBHOOK_QUERY_TOKEN>`  
3. `Update` 後に `Verify` を実行する。  
4. `Use webhook` を `Enabled` にする。  
5. テストメッセージ送信後、自営サーバログまたは GAS 実行ログで受信を確認する。  

チェックポイント:

- `https` 必須（`http` 不可）  
- `localhost`、社内IP、`127.0.0.1` は設定不可  
- 失敗時は `GET /health` の外部疎通、トンネル向き先ポート（`8000`）、チャネルシークレット整合を確認

## 7. 非機能要件

| 項目 | 内容 |
|------|------|
| 応答時間 | **エンドツーエンド**で目安 5〜30 秒（LLM・RAG が主因）。**LINE replyToken の制約**を満たすこと |
| 可用性 | **自営サーバ 1 台が SPOF**（単一障害点）。UPS・電源・NIC・ディスクの監視、**RPO/RTO** を運用で定義。冗長化は当面不要でもよいがリスクを認識する |
| ログ保持 | 自営サーバのディスク上にトラブル調査用ログ。ローテーション・保持期間は法務・個人情報方針に従う |
| セキュリティ | **LINE チャネルシークレット・アクセストークン・LLM API キー**は OS の環境変数、暗号化設定ファイル、または OS 標準のシークレット保管に限定。Webhook 以外の管理ポートは社内 LAN のみ or 強い認証。OS・ランタイムの**セキュリティ更新**を定期適用 |
| バックアップ | インデックス・設定・チャンク元データの**定期バックアップ**（外付けディスク、NAS、許可されたクラウドのいずれか） |
| 負荷 | 同時利用者が増える場合は **自営サーバのスペック**とアプリ内レート制限で対応（GAS クォータは対象外） |

## 8. コンプライアンス・運用

- 風営法等に関する**禁止表現・注意表現**は一覧化し、プロンプトおよび後処理で反映する。
- メッセージ履歴の利用目的・取得方法の**社内同意・委託契約**を整理する。
- 生成文は**必ず人間が最終確認**してから送信する運用を前提とする。

## 9. 用語

| 用語 | 説明 |
|------|------|
| RAG | Retrieval-Augmented Generation。検索した履歴チャンクを根拠として LLM が生成する方式（本構成では**自営サーバ上のアプリ**で実行） |
| 自営サーバ | オフィス内で **24 時間稼働**させる常設 PC。Webhook・RAG・LLM・`replyMessage` の実行主体 |
| GAS | Google Apps Script。**本仕様の必須構成ではない**。§3.5 のとおりトンネル不可時の**中継**に限定して任意利用可 |
| チャンク | 検索・埋め込みの最小単位となるテキストブロック |
| メッセージ履歴 | エクスポート JSON 等から取得した DM／メッセージの集合 |
